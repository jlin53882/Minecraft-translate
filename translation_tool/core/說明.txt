我將處理邏輯分為兩類來解釋：

1. 📂 路徑重複的處理 (檔案命名衝突)
程式的設計是將簡體中文的檔案路徑轉換為繁體中文的路徑，因此，它不會產生路徑衝突。

處理流程：
輸入檔案路徑 (簡體)：例如 assets/modid/_zh_cn/doc/intro.md

輸出檔案路徑 (繁體)：

程式會將 _zh_cn/ 替換為 _zh_tw/，生成輸出路徑：output_dir/assets/modid/_zh_tw/doc/intro.md。

寫入：

程式會先創建必要的目錄。

然後，它會將處理（S2TW 轉換或直接複製）後的內容寫入到這個特定的繁體路徑。

結論： 只要 ZIP 檔案內沒有同時存在 _zh_cn/ 和 _zh_tw/ 且名稱相同的檔案，就不會因為路徑轉換而導致衝突。

2. 📝 檔案內容重複的處理 (更新或覆蓋)
如果您的 ZIP 檔案是同一個檔案（例如：在不同批次處理時，或 ZIP 內有重複的檔案），處理邏輯如下：

A. 語言 JSON 檔案 (/lang/*.json) 的處理：
JSON 語言檔案是唯一採用增量更新和保留手動修改邏輯的檔案類型。

機制：

程式會檢查輸出目錄中是否已經存在上次處理的 zh_tw.json。

如果存在，它會將現有輸出檔案的內容作為基礎數據。

它只會將 zh_cn.json 中獨有的新鍵值合併進來。

結果：重複處理同一個語言模組時，已經翻譯過或手動修改過的條目會被保留，而 ZIP 內新的簡體條目才會被加入。

B. 內容檔案 (.md, .png, .snbt, .json 等) 的處理：
對於 _zh_cn/ 下的內容檔案，邏輯是簡單覆蓋。

機制：

程式處理完檔案（例如 S2TW 轉換 .md 檔案 或直接複製 .png 檔案）後，會使用 open(final_output_path, ...) 直接寫入。

結果：如果 output_dir 中已經存在一個舊的 _zh_tw/ 檔案，本次處理產生的新檔案內容會直接覆蓋掉舊檔案。
