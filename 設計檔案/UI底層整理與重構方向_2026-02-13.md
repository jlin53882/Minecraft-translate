## 2. UI 底層目前的「可重構點」

### 2.1 main.py 有一個 `create_view_wrapper()`
`main.py` 目前用 `create_view_wrapper()` 統一每個 View 的外觀（卡片、padding、陰影、margin）。

**問題：**
- 這個 wrapper 目前寫在 `main.py`，導致樣式規則難集中管理。
- 若要調整全站 UI（一致性、Dark mode、spacing），會一直回到 main 改。

**可重構方向（不碰被停用的四頁）：**
- 把 wrapper 抽到 `app/ui/` 或 `app/views/_shared/`（依專案現況選一個最少改動的落點）。
- main.py 只保留「組頁、導航、視窗尺寸」；樣式由共用 UI 模組提供。

### 2.2 CacheView 已經是一個很大頁（而且很關鍵）
CacheView 目前是「穩定版：總覽/管理 + 查詢頁入口（第一階段）」。

**注意：**
- 雖然 CacheView 裡出現「查詢頁入口」字樣，但它目前是入口/占位，且你的規則是「查詢不要設計」。
- 所以我們只能做：
  - **不改查詢頁本體**
  - 如果需要調整 CacheView 的 UI 底層（共用元件/樣式/註解/結構），只能做「不改功能、不推進查詢頁進度」的安全整理。

## 3. 建議的 UI 底層重構順序（針對本專案）

### Step 1（最安全）：抽出共用 Wrapper + 共用小元件
目標：
- 不動核心翻譯邏輯（translation_tool / services）
- 不碰被停用的四頁
- 先把 UI 一致性工具抽出來，降低後續改動成本

交付物：
- 例如：`app/ui/view_wrapper.py`（名稱可依你偏好調整）
- 提供：`wrap_view(content: ft.Control) -> ft.Container`
- 同步把 wrapper 的參數（padding/margin/radius/shadow/bgcolor）集中成常數，並加註解說明設計意圖。

### Step 2（可控）：為每個啟用頁補齊「中文註解 + 結構分段」
目標：
- 不做大改功能，只做「可讀性提升」與「拆小函式」
- 讓未來重寫/維護更輕鬆

### Step 3（進一步）：把大型 View 拆成 Presenter/Controller（可選）
你目前的程式風格其實混合了兩種：
- TranslationView：偏「View 自己管理狀態與事件」
- CacheView：看起來偏「更工程化的大頁」

若要統一，我建議以「對你最省力」為主：
- 先不強推 MVP/MVVM 全面導入
- 只對最痛的頁（通常是 CacheView / TranslationView）做局部整理
