# ITERATION_SOP.md
# Claw 疊代作業標準流程

版本：v1.3
適用範圍：所有 OpenClaw 疊代任務
狀態：repo 根目錄正式版；OpenClaw workspace 應與此檔同步

---

## 總原則

1. **不得自行宣稱完成** — 所有「完成」判斷必須有實際指令輸出為依據。
2. **不得捏造輸出** — Test result 區塊必須貼真實 terminal 輸出，不得改寫、選擇性省略錯誤，或把失敗寫成成功。
3. **兩個 Phase 必須分開執行** — Phase 1 完成後，先列出 `## Phase 1 完成清單`，再進入 Phase 2；不可把實作、驗證、PR 一次混寫成完成。
4. **沒有 Validation checklist 就停下來** — 若設計文件缺少 `## Validation checklist` 區塊，Claw 必須停止並要求補上，不得自行決定驗證範圍。
5. **設計稿與 PR 文件一律用 `.md` 保存** — 設計稿、設計討論整理稿、PR 文件，都應整理成 `.md` 檔，方便後續討論與追溯。

---

## 必須停下來問的情況
以下情況 Claw 不得自行把現況當契約，必須先問：
1. 不是單純 refactor，而是在定義什麼叫正確行為
2. 測試是在保舊行為，但無法確定那是不是 bug
3. caller 很多，但要刪的是 façade / shim / re-export
4. 某個舊行為看起來醜，但不確定是不是外部依賴
5. 設計稿沒有明講「這個舊行為要保還是要改」

## Phase 1：執行任務

### 1.1 讀取設計文件
- 找到本次任務對應的設計文件。
- 確認目標、範圍、以及明確列出的「不做的事」。
- 確認設計文件內已包含 `## Validation checklist`；若沒有，停下來要求補上。

### 1.2 執行
- 依設計文件逐項執行。
- 每完成一個子項目，內部記錄：已做 / 未做 / 無法做（原因）。
- 若涉及修改檔案，先建立可回退備份，再繼續下一輪修改。

### 1.3 Phase 1 完成檢查點
執行完畢後，列出以下清單後停下，**等待進入 Phase 2**：

```markdown
## Phase 1 完成清單
- [x] 做了：<項目>
- [x] 做了：<項目>
- [ ] 未做：<項目>（原因：）
- [ ] 超出範圍：<項目>
```

---

## Phase 2：驗證 + 寫 PR

### 2.1 執行 Validation Checklist
設計文件中必須包含 `## Validation checklist` 區塊。Claw 必須逐一執行，並擷取完整輸出。

格式範例：

```markdown
## Validation checklist
- [ ] `uv run pytest`
- [ ] `python -c "import app.views.cache_controller"`
- [ ] <其他驗證指令>
```

**規則：**
- 每條指令都必須實際跑，不得跳過。
- 輸出必須完整貼上，包含錯誤訊息。
- 若驗證失敗，必須記錄失敗原因，不得隱藏。
- 若某條指令無法執行，必須明確說明原因，不得偽裝成成功。

---

### 2.2 寫 PR / 設計文件
驗證完成後，依 template 產出文件。

**儲存路徑：** `docs/pr/YYYY-MM-DD_HHmm_PR_<topic>.md`

這條同時適用於：
- 正式 PR 文件
- 設計稿
- 設計討論整理稿

### 2.3 Rejected approaches 欄位標準（強制）
每份設計稿 / 設計討論稿 / 最終 PR 文件，都必須包含 `## Rejected approaches` 區塊，且固定放在 `## Not included in this PR` 前面。

固定格式：

```markdown
## Rejected approaches
- 試過：<做了什麼>
- 為什麼放棄：<哪個 test / 錯誤訊息 / 風險擋下來>
- 最終改採：<替代方案>（若有）
```

**強制要求：**
- 每份設計稿至少要有一條 Rejected approach。
- 若真的沒有嘗試過其他方案，不得留空；必須明確寫：
  - `試過：只考慮過一個方案`
  - `為什麼放棄：未嘗試其他方案，理由是：<為什麼此題只有一個合理方案>`
  - `最終改採：無`
- 這個欄位的目的，是強迫在 Phase 1 前先思考：
  - 有沒有更好的做法？
  - 有沒有看似可行但風險更高的方案？
  - 最後為什麼選現在這條路？
- 不得只寫空泛句子，例如「沒有其他方案」而不說理由。
- 若曾試過別的方案卻沒寫，視為文件不完整。

### 2.4 刪除/移除/替換說明標準（強制）
凡 PR 涉及以下任一動作：
- 刪除 function / class / method / field
- 刪除 import / module / package / file
- 移除 wrapper / façade / compatibility shim
- 停用既有流程並改走替代路徑
- 將某段舊邏輯替換為新實作

PR 文件中必須對**每一個刪除項目**逐項補上以下六個欄位，且**固定放在 `## Phase 1 完成清單` 後面**：

```markdown
### 刪除項目：`<名稱>`

- **為什麼改**：（這段為什麼存在、原本的意圖是什麼）
- **為什麼能刪**：（確認可刪的依據，例如：全域 grep 無引用、已有替代品）
- **目前誰在用 / 沒人在用**：（引用點清單，或確認無引用的方式）
- **替代路徑是什麼**：（如果有替代，替代在哪；如果沒有，說明不需要替代）
- **風險是什麼**：（刪錯或漏改的後果、有無 runtime 爆點）
- **我是怎麼驗證的**：（用什麼指令、輸出是什麼）
```

**強制要求：**
- 若任一欄位資訊不足，必須明確標記 `[需確認]`。
- 不得用「應該」「看起來」「大概沒人用」這類模糊語句帶過。
- 若聲稱「無引用」，必須附實際搜尋方式或指令。
- 若聲稱「已有替代」，必須寫出替代路徑與現行 caller。
- 若本 PR 沒有刪除項目，可省略這個章節；若有，不得改放其他位置。

## 違規情境與處理

| 情境 | 錯誤行為 | 正確行為 |
|------|----------|----------|
| 測試有錯誤 | 省略錯誤，寫「測試通過」 | 完整貼出錯誤，標記為既有問題或新問題 |
| 某項目沒做到 | 在 PR 中略過不提 | 列入「Not included in this PR」 |
| 驗證指令無法執行 | 寫「已驗證」 | 說明無法執行的原因，不得偽裝成成功 |
| 設計文件沒有 Validation checklist | 跳過驗證 | 停下來，通知 James 補上 checklist 後再繼續 |
| Rejected approaches 留空 / 缺漏 | 直接略過方案比較 | 至少寫一條，若只考慮一個方案也要明確寫理由 |
| 有刪除項目但未寫刪除理由 | 只寫「已移除」 | 逐項補上六欄位刪除說明 |
