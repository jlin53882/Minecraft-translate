## 刪除/移除/替換說明（若有，固定放這裡）

### 刪除項目：`app/services.py` 內 inline `run_ftb_translation_service()` 實作
- **為什麼改**：PR22 的目標是把 `app/services.py` 內的 FTB pipeline wrapper 抽離到 `services_impl/pipelines/`，讓 `services.py` 更接近單純 façade，避免繼續承載各種 pipeline 細節。
- **為什麼能刪**：原本實作已搬到 `app/services_impl/pipelines/ftb_service.py`，而且 `app/services.py` 仍保留同名 re-export，所以既有 caller 不需要改。
- **目前誰在用 / 沒人在用**：實際 caller 為 `app/views/translation_view.py` 的 lazy import 區塊，以及 `_run_ftb()` 內呼叫 `run_ftb_translation_service(...)`。沒有在 `main.py` / `tests` 看到其他直接 caller。
- **替代路徑是什麼**：實作改由 `app/services_impl/pipelines/ftb_service.py::run_ftb_translation_service` 提供；對外入口仍由 `app/services.py` re-export。
- **風險是什麼**：若 re-export 漏掉，`translation_view.py` 的 lazy import 會失敗，導致 FTB service 變成 `None`；若搬移時 step flags、`dry_run` 或 `write_new_cache` 傳遞錯位，FTB 行為會回歸；若 error handling 或 `UI_LOG_HANDLER.set_session(None)` 漏掉，任務狀態會異常。
- **我是怎麼驗證的**：執行 `uv run python -c "from app.services import run_ftb_translation_service"`、`uv run python -c "from app.views.translation_view import TranslationView; print('ok')"`、`uv run pytest -q tests/test_ui_refactor_guard.py`，均通過。

### 刪除項目：`app/services.py` 內 `translate_directory_generator` 直接 import
- **為什麼改**：這個 import 原本就不是 FTB wrapper 真正使用的入口；FTB service 內實際用的是函式內部 import 的 `run_ftb_pipeline`。在 PR22 搬移後，`app/services.py` 已不需要保留這個殘留 import。
- **為什麼能刪**：`app/services.py` 已無直接使用；搜尋 `app/services.py app/views/translation_view.py tests main.py` 後，也沒有其他地方透過 `services.py` 依賴這個符號。
- **目前誰在用 / 沒人在用**：`app/services.py` 已無使用；`translation_tool.core.ftb_translator` 仍由 `ftb_service.py` 在 runtime 透過 `run_ftb_pipeline` 使用。
- **替代路徑是什麼**：不再需要 façade 層持有這個殘留 import；實際 FTB 執行入口仍在 `ftb_service.py` 裡的 `run_ftb_pipeline`。
- **風險是什麼**：若誤判這個 import 仍被依賴，可能造成 import fail；但目前實查沒有 caller，風險低。
- **我是怎麼驗證的**：`uv run python -c "from app.services_impl.pipelines import ftb_service"`、`uv run python -c "from app.services import run_ftb_translation_service"`、`uv run python -c "from app.views.translation_view import TranslationView; print('ok')"` 均通過。

## What was done

### 1. 新增 `app/services_impl/pipelines/ftb_service.py`
新增 FTB 專用 pipeline service 模組，將原本的 `run_ftb_translation_service()` 搬入。

這次保留不變的點：
- 函式簽名不變
- `dry_run` / `step_export` / `step_clean` / `step_translate` / `step_inject` / `write_new_cache` 參數與順序不變
- 仍在 service 內呼叫 `run_ftb_pipeline(...)`
- `session.start()` / `session.finish()` / `session.set_error()` 行為不變
- `UI_LOG_HANDLER.set_session(None)` 行為不變

### 2. 新模組內自行接回 logger config 更新責任
原本 FTB wrapper 直接呼叫 `services.py::update_logger_config()`。
搬移後，新的 `ftb_service.py` 透過：
- `app.services_impl.config_service._load_app_config`
- `app.services_impl.logging_service.update_logger_config(...)`

在模組內建立 `_update_logger_config()`，維持每次任務開始前都重新套用 logger config 的行為。

這樣做的理由：
- 避免 `ftb_service.py` 反向 import `app.services` 造成 circular import
- 保住原本 runtime 行為，不改 caller

### 3. `app/services.py` 改為 façade / re-export
`app/services.py` 的 FTB 區塊已改成：
- 刪除 inline `run_ftb_translation_service()`
- 刪除不再需要的 `translate_directory_generator` import
- 新增：
  - `from app.services_impl.pipelines.ftb_service import run_ftb_translation_service`

效果是：
- `TranslationView` 不需要改 import
- lazy import fallback 邏輯維持原樣
- 本顆 PR 只改 service 實作位置，不動 UI
- `services.py` 繼續往薄 façade 收斂

### 4. 這顆 PR 刻意不碰 `ftb_translator.py`
這次**沒有**修改 `translation_tool/core/ftb_translator.py`。
原因：
- PR22 的目標是 service split，不是 core FTB refactor
- 如果這顆 PR 同時改 wrapper 與 core pipeline，問題一出來很難切責任
- FTB 本身已經有多個 step flags 與 pipeline 參數，應避免 scope 膨脹

### 5. 備份位置
本次修改前已建立可回退備份：
- `backups/pr22-ftb-split-20260311-2340/app/services.py`

依目前專案備份規則，備份統一放在 `backups/<pr-slug-timestamp>/`，並保留原相對路徑。

## Important findings
- FTB 真正敏感的是參數邊界與 lazy import fallback，不是檔案搬移本身。
- `translation_view.py` 採 `try/except` lazy import，所以驗證時一定要直接做 view import smoke，單看 `from app.services import ...` 不夠。
- 這顆 PR 可以安全拆的前提，依然是只搬 service，不碰 `TranslationView` 內部 UI orchestration。
