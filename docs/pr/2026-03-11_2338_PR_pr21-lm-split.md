## 刪除/移除/替換說明（若有，固定放這裡）

### 刪除項目：`app/services.py` 內 inline `run_lm_translation_service()` 實作
- **為什麼改**：PR21 的目標是把 `app/services.py` 內的 LM pipeline wrapper 抽離到 `services_impl/pipelines/`，讓 `services.py` 更像單純 façade，而不是所有 pipeline 實作的堆放處。
- **為什麼能刪**：原本實作已搬到 `app/services_impl/pipelines/lm_service.py`，而且 `app/services.py` 仍保留同名 re-export，所以既有 caller 不需要改。
- **目前誰在用 / 沒人在用**：實際 caller 為 `app/views/lm_view.py` 第 10 行 import，以及 `start_clicked()` 內背景 thread 直接以 `target=run_lm_translation_service` 啟動。沒有在 `main.py` / `tests` 看到其他直接 caller。
- **替代路徑是什麼**：實作改由 `app/services_impl/pipelines/lm_service.py::run_lm_translation_service` 提供；對外入口仍由 `app/services.py` re-export。
- **風險是什麼**：若 re-export 漏掉，`LMView` 會 import fail；若搬移時漏掉 `dry_run` 前後提示 log、`write_new_cache` / `export_lang` 參數傳遞或 `GLOBAL_LOG_LIMITER.flush()`，LM 行為會回歸；若 `UI_LOG_HANDLER.set_session(None)` 漏掉，後續任務可能吃到舊 session。
- **我是怎麼驗證的**：執行 `uv run python -c "from app.services import run_lm_translation_service"`、`uv run python -c "from app.views.lm_view import LMView; print('ok')"`、`uv run pytest -q tests/test_ui_refactor_guard.py`，均通過。

### 刪除項目：`app/services.py` 內 `lm_translate_gen` 直接 import
- **為什麼改**：LM generator 的直接使用點已從 `app/services.py` 轉移到 `app/services_impl/pipelines/lm_service.py`，façade 層不應再直接依賴它。
- **為什麼能刪**：`app/services.py` 已無直接使用；新的唯一使用點在 `lm_service.py`。
- **目前誰在用 / 沒人在用**：`app/services.py` 已無使用；`app/services_impl/pipelines/lm_service.py` 仍使用中。
- **替代路徑是什麼**：改由 `lm_service.py` 匯入並呼叫 `translate_directory_generator`。
- **風險是什麼**：如果新模組沒正確匯入 LM generator，LM service import 會失敗或 runtime fail。
- **我是怎麼驗證的**：`uv run python -c "from app.services_impl.pipelines import lm_service"` 與 `uv run python -c "from app.services import run_lm_translation_service"` 均通過。

## What was done

### 1. 新增 `app/services_impl/pipelines/lm_service.py`
新增 LM 專用 pipeline service 模組，將原本的 `run_lm_translation_service()` 搬入。

這次保留不變的點：
- 函式簽名不變
- `dry_run` 前後提示 log 保持不變
- `export_lang` / `write_new_cache` 原樣傳給 LM generator
- `update_dict` 轉 `session.add_log()` / `session.set_progress()` / `session.set_error()` 的流程不變
- generator 結束後仍做 `GLOBAL_LOG_LIMITER.flush()`
- `UI_LOG_HANDLER.set_session(None)` 行為不變

### 2. 新模組內自行接回 logger config 更新責任
原本 LM wrapper 直接呼叫 `services.py::update_logger_config()`。
搬移後，新的 `lm_service.py` 透過：
- `app.services_impl.config_service._load_app_config`
- `app.services_impl.logging_service.update_logger_config(...)`

在模組內建立 `_update_logger_config()`，維持每次任務開始前都重新套用 logger config 的行為。

這樣做的理由：
- 避免 `lm_service.py` 反向 import `app.services` 造成 circular import
- 保住原本 runtime 行為，不改 caller

### 3. `app/services.py` 改為 façade / re-export
`app/services.py` 的 LM 區塊已改成：
- 刪除 inline `run_lm_translation_service()`
- 刪除不再需要的 `lm_translate_gen` import
- 新增：
  - `from app.services_impl.pipelines.lm_service import run_lm_translation_service`

效果是：
- `LMView` 不需要改 import
- 本顆 PR 只改 service 實作位置，不動 UI
- `services.py` 繼續往薄 façade 收斂

### 4. 這顆 PR 刻意不碰 `lm_translator.py`
這次**沒有**修改 `translation_tool/core/lm_translator.py`。
原因：
- PR21 的目標是 service split，不是 core LM refactor
- 如果這顆 PR 同時改 wrapper 與 core translator，問題一出來很難切責任
- LM pipeline 已經有 `dry_run` / cache / 參數傳遞等敏感點，應避免 scope 膨脹

### 5. 備份位置
本次修改前已建立可回退備份：
- `backups/pr21-lm-split-20260311-2335/app/services.py`

依目前專案備份規則，備份統一放在 `backups/<pr-slug-timestamp>/`，並保留原相對路徑。

## Important findings
- LM service 雖然只有一顆，但它的行為邊界比 extract / bundle 更敏感，因為同時涉及 `dry_run`、cache 參數、flush 與 session lifecycle。
- 把 logger config 更新責任內收進 `lm_service.py` 是必要的，不然就得反向 import `app.services`，很容易做出 circular import。
- 這顆 PR 的核心原則依然是：只搬位置，不改行為。
