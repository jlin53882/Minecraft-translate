## 刪除/移除/替換說明（若有，固定放這裡）

### 刪除項目：`app/services.py` 內 inline `run_merge_zip_batch_service()` 實作
- **為什麼改**：PR20 的目標是繼續把 `app/services.py` 往 façade 層收斂。merge 雖然只有一顆 function，但它屬於明確的 pipeline service，應該和 lookup / bundle / extract 一樣搬進 `services_impl/pipelines/`。
- **為什麼能刪**：原本實作已搬到 `app/services_impl/pipelines/merge_service.py`，而且 `app/services.py` 仍保留同名 re-export，所以既有 caller 不需要改。
- **目前誰在用 / 沒人在用**：實際 caller 為 `app/views/merge_view.py` 第 10 行 import，以及 `start_merge()` 內背景 thread 直接以 `target=run_merge_zip_batch_service` 啟動。沒有在 `main.py` / `tests` 看到其他直接 caller。
- **替代路徑是什麼**：實作改由 `app/services_impl/pipelines/merge_service.py::run_merge_zip_batch_service` 提供；對外入口仍由 `app/services.py` re-export。
- **風險是什麼**：若 re-export 漏掉，`MergeView` 會 import fail；若搬移時把 inner generator 沒有正確 iterate，merge 實際上不會執行；若 progress 疊加公式變動，UI 進度體感會回歸；若 nested try/except 邏輯被改壞，錯誤資訊可能丟失。
- **我是怎麼驗證的**：執行 `uv run python -c "from app.services import run_merge_zip_batch_service"`、`uv run python -c "from app.views.merge_view import MergeView; print('ok')"`、`uv run pytest -q tests/test_ui_refactor_guard.py`，均通過。

### 刪除項目：`app/services.py` 內 `merge_zhcn_to_zhtw_from_zip` 直接 import
- **為什麼改**：merge 的直接使用點已從 `app/services.py` 轉移到 `app/services_impl/pipelines/merge_service.py`，façade 層不應再直接依賴 core merge generator。
- **為什麼能刪**：`app/services.py` 已無直接使用；新的唯一使用點在 `merge_service.py`。
- **目前誰在用 / 沒人在用**：`app/services.py` 已無使用；`app/services_impl/pipelines/merge_service.py` 仍使用中。
- **替代路徑是什麼**：改由 `merge_service.py` 匯入並呼叫。
- **風險是什麼**：如果新模組沒正確匯入 `merge_zhcn_to_zhtw_from_zip`，merge service import 會失敗或 runtime fail。
- **我是怎麼驗證的**：`uv run python -c "from app.services_impl.pipelines import merge_service"` 與 `uv run python -c "from app.services import run_merge_zip_batch_service"` 均通過。

## What was done

### 1. 新增 `app/services_impl/pipelines/merge_service.py`
新增 merge 專用 pipeline service 模組，將原本的 `run_merge_zip_batch_service()` 原樣搬入。

這次保留不變的點：
- 函式簽名不變
- ZIP 外層 progress + inner generator progress 疊加公式不變
- `merge_zhcn_to_zhtw_from_zip(...)` 仍被明確 iterate
- empty ZIP list 仍直接 `session.finish()`
- 每顆 ZIP 完成後仍額外推進一次 `session.set_progress((idx + 1) / total)`
- `UI_LOG_HANDLER.set_session(None)` 行為不變

### 2. 新模組內自行接回 logger config 更新責任
原本 merge wrapper 直接呼叫 `services.py::update_logger_config()`。
搬移後，新的 `merge_service.py` 透過：
- `app.services_impl.config_service._load_app_config`
- `app.services_impl.logging_service.update_logger_config(...)`

在模組內建立 `_update_logger_config()`，維持每次任務開始前都重新套用 logger config 的行為。

這樣做的理由：
- 避免 `merge_service.py` 反向 import `app.services` 造成 circular import
- 保住原本 runtime 行為，不改 caller

### 3. `app/services.py` 改為 façade / re-export
`app/services.py` 的 merge 區塊已改成：
- 刪除 inline `run_merge_zip_batch_service()`
- 刪除不再需要的 `merge_zhcn_to_zhtw_from_zip` import
- 新增：
  - `from app.services_impl.pipelines.merge_service import run_merge_zip_batch_service`

效果是：
- `MergeView` 不需要改 import
- 本顆 PR 只改 service 實作位置，不動 UI
- `services.py` 繼續往薄 façade 收斂

### 4. 這顆 PR 刻意不碰 `lang_merger.py`
這次**沒有**修改 `translation_tool/core/lang_merger.py`。
原因：
- PR20 的目標是 service split，不是 core merge refactor
- 如果這顆 PR 同時改 wrapper 與 core merge，問題一出來很難切責任
- merge 本身 already 屬邏輯敏感區，應避免 scope 膨脹

### 5. 備份位置
本次修改前已建立可回退備份：
- `backups/pr20-merge-split-20260311-2330/app/services.py`

依目前專案備份規則，備份統一放在 `backups/<pr-slug-timestamp>/`，並保留原相對路徑。

## Important findings
- merge 真正敏感的不是檔案數量，而是 wrapper 內的「progress 疊加 + nested try/except + 必須 iterate generator」這三件事。
- 這顆 PR 可以安全拆的前提，就是只搬位置，不碰行為。
- 把 logger config 更新責任內收進 `merge_service.py` 是必要的，不然就得反向 import `app.services`，很容易做出 circular import。
